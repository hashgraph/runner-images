# Conventions

- **Apache 2.0 license headers on workflows.** Every file under `.github/workflows/` carries the `Copyright (C) <year> Hedera Hashgraph, LLC` Apache 2.0 header block on line 1. Copy the header from any existing workflow when adding a new one. This is an open-source repo — never substitute a proprietary/confidential header.
- **Keep the two Dockerfiles in sync.** `Dockerfile` (apt) and `Dockerfile.chainguard` (apk) install the same toolchain through different package managers. A change to one almost always needs the mirrored change in the other. The `## Begin/End ... Customizations ##` comment fences mark the regions that must stay aligned. Deliberate divergences (UID/GID: `docker` group is 123 in Debian, 321 in Chainguard; `ImageOS` env value; `add-apt-repository` git PPA which only applies to Ubuntu) are intentional — preserve them rather than unifying.
- **Pin tool versions with ARG + default.** New tools added to an image follow the existing pattern — `ARG TOOL_VERSION=x.y.z` with a default, consumed in the `RUN` that installs it, and (when CI should be able to override it) threaded through `build-args` in `zxc-build-scaleset-images.yaml`. Don't install unpinned `latest`.
- **GitHub Actions pinning.** Every external `uses:` in `.github/workflows/*` is pinned by full commit SHA, not by tag, with a `# vX.Y.Z` trailing comment for human readability. Dependabot keeps the SHA fresh. Never pin by floating tag.
- **GitHub Actions naming.** Filename prefix encodes the role: `flow-*` for entry points (PR / release / scheduled-scan), `zxc-*` for reusable `workflow_call` callees, `zxcron-*` for `schedule`-triggered, `zxf-*` for fork handling. Workflow boilerplate: line-1 Apache 2.0 header, `defaults.run.shell: bash`, explicit `permissions:` (least privilege — escalate per job/workflow), `concurrency:` with `cancel-in-progress: true` for PR-time flows, and `step-security/harden-runner` as the first step of every job.
- **Conventional Commit titles on PRs.** PR titles must match Conventional Commits — enforced by `flow-pull-request-formatting.yaml` and locally by the `conventional-pre-commit` commit-msg hook. Release-bot commits use `chore(release): scaleset-v<x> [skip ci]`; dependency bumps use `chore(deps): ...` / `build(deps): ...`. Squash-merge is the default.
- **Sign every commit, two ways — mandatory.** Every contributor commit MUST be (1) GPG-signed (`git config commit.gpgsign true`, `user.signingkey <KEY-ID>`; `%G?` shows `G` on `git log --show-signature`) and (2) carry a DCO `Signed-off-by:` trailer (auto-appended by the `prepare-commit-msg` hook documented in `.claude/git-hooks.md`). Both are required on a fresh clone before committing. Never bypass either with `--no-gpg-sign`, `--no-verify`, or by hand-removing the trailer.
- **Version descriptor is machine-owned.** `scaleset/runner/VERSION` is updated by the release flow, not by hand. Editing it manually outside a release is drift.
- **Tool cache is a build artifact.** `scaleset/runner/tools/` is filled in during the CI build and `COPY`d into the image — never commit its contents.
- **Secret hygiene.** No plaintext credentials in the tree, ever — `gitleaks` runs in pre-commit and CI. Image builds authenticate via the workflow `GITHUB_TOKEN` (GHCR) and JFrog OIDC (Artifactory); don't introduce static tokens.
- **Forks are not trusted.** Contributions from forked repositories are auto-closed (`zxf-forked-pull-request-closer.yaml`); fork-sensitive workflows guard with `if: ${{ !github.event.pull_request.head.repo.fork }}`. Keep that guard when adding workflows that touch secrets.
- **Don't push or release on the user's behalf.** Building images locally and dry-running is fine; pushing to a registry, dispatching the release workflow, or cutting a `scaleset-v*` release is a user-driven action only.
